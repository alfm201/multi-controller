# multi-controller 장기 로드맵

## 1. 문서 목적

이 문서는 `multi-controller`의 앞으로의 방향과 우선순위만 정리하는 장기 로드맵이다.

- 완료 기록, 차수별 회고, `1차 / 2차 / 3차` 같은 과거 진행 구분은 남기지 않는다.
- 각 항목은 `Now / Next / Later` 우선순위와 완료 기준을 가진다.
- 세부 구현 계획은 별도의 `agent/PLAN.md`에서 다루고, 이 문서는 제품 방향과 장기 구조를 정리하는 데 집중한다.

## 2. 제품 방향

이 프로젝트의 목표는 여러 Windows PC를 하나의 자연스러운 작업 공간처럼 느끼게 만드는 것이다.

- 사용자는 PC 경계와 제어권 전환을 의식하기보다, 작업 공간이 이어진다고 느껴야 한다.
- GUI는 운영자용 디버그 패널이 아니라 실제 사용 중 계속 열어둘 수 있는 제어 패널이어야 한다.
- 자동 전환, 레이아웃, 모니터 맵 기능은 유지하되, 조작 방식은 텍스트 입력보다 직접 조작 중심으로 이동한다.
- 모니터 정보는 장기적으로 `OS 자동 인식 + 사용자 보정` 흐름으로 간다.
- GUI 프레임워크는 계속 Tkinter를 사용하되, 기본 Windows 위젯 느낌은 줄이고 프로젝트 고유의 시각 시스템을 입힌다.

## 3. 현재 핵심 문제

현재 사용자 경험에서 가장 크게 드러나는 문제는 아래와 같다.

1. `runtime/status_window.py`와 `runtime/layout_editor.py`의 주기적 전체 갱신 때문에 깜빡임과 잔렉이 있다.
2. `tk`와 `ttk`를 섞어 쓰는 현재 방식 때문에 배경색과 카드/배지/버튼 톤이 어색하게 섞인다.
3. 레이아웃 탭에 모니터 preset 버튼까지 들어와 있어 액션 밀도가 높고 역할이 겹친다.
4. 버튼과 체크박스가 기본 Windows 위젯 느낌이라 전체 UI가 구시대적으로 보인다.
5. 모니터 맵 편집기는 클릭 선택 + 고정 행/열 확장 방식이라 조작 흐름이 끊긴다.
6. 실제 연결된 모니터 정보를 런타임이 모르기 때문에 사용자가 수동으로 맞춰야 할 부분이 많다.

## 4. 장기 로드맵

### Track A. UI 성능 및 렌더링 안정화 (`Now`)

목표:
- 상태창의 깜빡임과 잔렉을 줄이고, 레이아웃 편집 중 redraw 비용을 낮춘다.

핵심 방향:
- 상태창 refresh를 `전체 재구성`에서 `dirty update` 기반으로 전환한다.
- 레이아웃 캔버스는 `layout / selection / viewport / online state`가 바뀔 때만 다시 그린다.
- 요약 카드, 빠른 전환, 연결 상태 표, 고급 정보는 위젯을 재사용하고 값만 갱신한다.
- 현재 500ms polling은 유지하되, polling과 repaint를 분리한다.

완료 기준:
- idle 상태에서 눈에 띄는 깜빡임이 없다.
- drag / pan / zoom 중 프레임 드랍 체감이 크게 줄어든다.
- refresh tick이 있어도 카드, 표, 상세 패널이 통째로 다시 만들어지지 않는다.

### Track B. 시각 시스템 현대화 (`Now -> Next`)

목표:
- UI를 더 현대적이고 일관된 화면으로 정리하고, 기본 Windows 위젯 느낌을 줄인다.

핵심 방향:
- 전역 `ttk.Style` 기반 테마 레이어를 도입하고 palette, spacing, border 규칙을 한곳에서 관리한다.
- 회색 바탕을 기본으로 깔지 않고, `밝은 중성 배경 + 선명한 카드/배지 대비`로 재정리한다.
- 기본 Windows 체크박스/버튼을 그대로 쓰는 위치는 `2-state 버튼`, `segmented control`, `badge` 스타일로 대체한다.
- `tk.Button`과 `tk.Label`을 시각 장식용으로 쓰는 부분은 가능한 한 통일된 컴포넌트 계층으로 정리한다.

완료 기준:
- 체크박스 glyph와 구형 기본 버튼 느낌이 주요 화면에서 사라진다.
- 배경, 카드, 배지, 툴바가 하나의 스타일 시스템처럼 보인다.
- 다크모드 대응은 이번 로드맵 범위 밖으로 둔다.

### Track C. 모니터 맵 편집 UX 재설계 (`Next`)

목표:
- 모니터 맵 편집을 클릭 위주 보정 UI에서 drag-and-drop 중심 직접 조작 UI로 바꾼다.

핵심 방향:
- 모니터 맵 편집기는 `drag-and-drop 우선`으로 바꾸고, 클릭은 보조 선택 동작으로 낮춘다.
- 보드는 고정 행/열 확장 대신 `가장자리 drag 시 자동 확장`, `drop 후 필요 시 trim` 흐름으로 바꾼다.
- `1xN / 2x2 / 3x2` preset은 레이아웃 탭에서 제거하고 모니터 맵 편집창에만 둔다.
- 논리/물리 배치 차이는 같은 display 색상과 hover / selection highlight로 즉시 비교되게 한다.

완료 기준:
- display를 끌어서 옮기는 동작만으로 주요 편집이 가능하다.
- 행/열 추가 버튼 의존도가 크게 줄어든다.
- preset은 모니터 맵 편집창 내부에서만 노출된다.

### Track D. 모니터 자동 인식과 동기화 (`Later`)

목표:
- 각 PC의 실제 연결 모니터 정보를 자동으로 읽고, 사용자는 물리 배치만 보정하도록 만든다.

핵심 방향:
- 목표는 `논리 배치 자동 수집 + 물리 배치 사용자 보정`이다.
- OS에서 모니터 수, device name, bounds, primary 여부, DPI, 논리 배치를 읽는 `read-only monitor inventory` 계층을 만든다.
- GUI는 자동 감지 결과를 보여주고, 사용자는 물리 배치만 보정해서 저장한다.
- 자동 감지 결과는 즉시 `config.json`을 덮어쓰지 않고, `검토 -> 적용 -> 저장` 흐름을 거친다.
- 장기적으로는 새 모니터 연결/해제 시 GUI에서 `감지 결과와 현재 설정 차이`를 보여준다.

완료 기준:
- 각 PC의 실제 연결 모니터 수와 논리 배치를 자동으로 인식한다.
- 사용자는 물리 배치만 보정하면 된다.
- 감지 결과와 수동 override의 책임 경계가 문서와 UI에 명확히 보인다.

## 5. 후순위 / 범위 밖 항목

아래 항목은 현재 장기 로드맵 범위에 포함하지 않는다.

- GUI 프레임워크 교체
- GPU 제어판 또는 Windows 디스플레이 설정을 앱이 직접 변경하는 기능
- 물리 배치 완전 자동 판정
- 테마 마켓 / 스킨 시스템
- 인터넷 경유 노드 탐지와 계정 기반 동기화

## 6. 작업 원칙

- 성능 문제는 `느낌`이 아니라 `불필요한 redraw 제거` 기준으로 해결한다.
- UI 정리는 버튼 수를 줄이는 것보다 `역할 위치를 맞추는 것`을 우선한다.
- 자동 인식은 논리 배치를 우선하고, 물리 배치는 사용자 확인 단계를 남긴다.
- `ROADMAP.md`에는 완료 기록을 남기지 않는다.
- 세부 구현은 가능하면 테스트와 함께 진행하고, 수동 검증 기준도 같이 정리한다.
