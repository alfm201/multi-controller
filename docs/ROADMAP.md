# multi-controller Roadmap

## 1. 문서 목적

이 문서는 앞으로 만들 일만 적는다.

- 완료 보고나 차수별 회고는 남기지 않는다.
- 지금 당장 실행할 일은 `agent/PLAN.md`에서 관리한다.
- 여기서는 `Now / Next / Later` 기준으로 중장기 방향만 정리한다.

## 2. 제품 방향

- 여러 PC가 하나의 자연스러운 작업 공간처럼 느껴져야 한다.
- GUI는 디버그 패널이 아니라 실사용 제어 패널이어야 한다.
- 실제 모니터 감지를 기본으로 삼고, 사용자는 필요한 물리 보정만 한다.
- 설정은 `config/` 아래에서 책임별로 나뉘고, 백업과 복구 경로가 명확해야 한다.
- tray, GUI, 로그는 서로 다른 역할을 맡되 같은 상태 언어를 사용해야 한다.

## 3. 다음 장기 과제

### Track A. 상태 메시지 표면 재설계 (`Now`)

목표:
- 화면 하단의 짧은 상태 문구를 더 알아보기 쉬운 운영 피드백 표면으로 바꾼다.

작업 방향:
- 성공/경고/실패/대기 상태를 배너, 인라인 힌트, 기록 가능한 알림으로 구분한다.
- 레이아웃 편집, 모니터 맵, 노드 관리, 재감지 요청이 같은 피드백 규칙을 쓰게 만든다.
- 사라지는 일회성 메시지와 계속 남아 있어야 하는 상태를 분리한다.

완료 기준:
- 사용자가 지금 막 일어난 일과 아직 기다려야 하는 일을 직관적으로 구분할 수 있다.

### Track B. 키보드/접근성 운영 UX (`Next`)

목표:
- 마우스 중심 GUI에 키보드 이동과 접근성 배려를 보강한다.

작업 방향:
- 주요 버튼, 탭, 리스트, 레이아웃 선택에 키보드 포커스 이동을 넣는다.
- 모니터 맵 편집에도 선택 이동, 배치, 되돌리기 단축키를 추가한다.
- 색상 외에도 텍스트와 아이콘으로 상태 차이를 충분히 구분한다.

완료 기준:
- 주요 운영 흐름을 키보드만으로도 무리 없이 따라갈 수 있다.

### Track C. 설치/복구 흐름 단순화 (`Next`)

목표:
- 새 환경에서 시작하거나 문제가 생겼을 때 복구하는 과정을 더 짧게 만든다.

작업 방향:
- `--init-config`, `--migrate-config`, 백업 복구를 하나의 운영 가이드로 묶는다.
- 예제 config와 실제 설치 경로 예시를 더 명확히 제공한다.
- 복구 가능한 백업 목록 보기, 선택 복구, 백업 정리 같은 보조 도구를 검토한다.

완료 기준:
- 처음 설치하는 사람도 설정 생성, 분리, 복구 경로를 문서만 보고 따라갈 수 있다.

### Track D. 운영 진단 타임라인 (`Later`)

목표:
- tray, GUI, 로그를 넘나들며 최근에 무슨 일이 있었는지 한눈에 볼 수 있게 만든다.

작업 방향:
- target 전환, lease 변화, stale inventory, 모니터 diff, 노드 수정 같은 운영 이벤트를 짧은 타임라인으로 정리한다.
- GUI와 로그가 같은 사건 이름과 심각도 체계를 공유하게 만든다.

완료 기준:
- 최근 운영 이슈를 GUI만으로도 빠르게 되짚을 수 있다.

## 4. 후순위 / 범위 밖 항목

- GUI 프레임워크 교체
- OS 디스플레이 설정을 앱이 직접 변경하는 기능
- 물리 배치 완전 자동 판정
- 인터넷 경유 노드 탐지와 계정 기반 동기화
- 대규모 테마 시스템

## 5. 작업 원칙

- 성능 문제는 redraw 빈도와 UI thread 점유 시간 기준으로 판단한다.
- 실제 감지 정보가 우선이고, 수동 보정은 필요한 만큼만 남긴다.
- 설정 파일 분리는 파일 수보다 책임 경계가 명확한지가 더 중요하다.
- `ROADMAP.md`에는 완료 보고를 적지 않는다.
