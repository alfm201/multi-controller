# Multi Screen Pass Roadmap

## 1. 문서 목적

이 문서는 현재 코드베이스 기준으로 앞으로 다룰 엔지니어링 과제를 정리한다.

- 완료 보고나 과거 phase 기록은 최소화한다.
- 우선순위는 `Now / Next / Later`로만 관리한다.
- 새 기능 추가보다 구조, 안정성, 성능, 운영 경험을 먼저 다룬다.

## 2. Now

### A. 자동 업데이트 정책 정리

목표:
- 이미 있는 업데이트 기능을 어떤 배포 정책과 UX로 운영할지 결정한다.

핵심 작업:
- 수동 확인 / 자동 확인 / 자동 설치 범위를 구분
- installer 기반 업데이트 UX와 실패 복구 흐름 점검
- 코드 서명과 신뢰 UX를 포함한 배포 정책 정리

완료 기준:
- 자동 업데이트를 기본 정책으로 둘지, 수동 설치 중심으로 유지할지와 그 근거가 정리된다.

## 3. Next

### A. UI/상태 화면 단순화 2차

목표:
- 작은 창에서도 읽기 쉽고, 필요한 정보만 보이게 만든다.

핵심 작업:
- 상태 카드, 노드 목록, 설정 화면 정보 밀도 재조정
- 중복 상태 표시와 긴 안내 문구 축소

완료 기준:
- 기본 창 크기가 작아도 주요 흐름을 읽고 조작하는 데 무리가 없다.

## 4. Later

### A. 입력 계층 재평가

목표:
- low-level hook 이후에도 더 낮은 입력 계층이 필요한지 판단한다.

핵심 작업:
- hook 기반 구조의 한계와 운영 리스크 정리
- raw input 또는 더 낮은 계층이 필요한 조건 문서화

### A-1. Windows 권한/UIPI + fullscreen 상호작용 점검

목표:
- 일반 권한 실행 시 fullscreen 앱 focus 상태에서 논리/물리 배치가 무시되거나, focus 해제 직후 첫 move가 edge warp로 이어지는 현상을 Windows 권한/UIPI 관점에서 다시 확인한다.

핵심 작업:
- 일반 권한 vs 관리자 권한에서 fullscreen 앱 focus 중 low-level hook, ClipCursor, SetCursorPos, actual pointer snapshot이 어떻게 달라지는지 재현 로그 확보
- fullscreen focus 중 배치 무시 현상이 capture 누락인지, stale routing context인지, ClipCursor 유지 실패인지 분리
- focus 해제 직후 첫 move warp가 repair warp 경로인지, stale admission context 경로인지 실기기에서 추가 확인
- 필요하면 Windows integrity/UIPI mismatch 진단을 사용자 노출 로그/진단 정보에 더 분명히 남길지 검토

완료 기준:
- fullscreen + 일반 권한 조합에서 남는 현상이 앱 내부 구조 문제인지 Windows 권한/UIPI 한계인지 구분해서 설명할 수 있다.
- 코드로 줄일 수 있는 부분과 운영/권한 안내로 남겨야 할 부분이 분리된다.

### B. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포, 복구, 실행 경로, 백업 정책을 운영 문서와 함께 정리한다.

핵심 작업:
- installer/portable 기준 문서화
- recovery watchdog / 수동 복구 / backup 정책 정리
- 포트 충돌 시 예외 처리 및 포트 풀 fallback 검토

### C. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠 이상 장시간 실행하는 환경에서 병목과 누적 문제를 줄인다.

핵심 작업:
- 입력 큐 적체 가능성 점검
- watcher / reporter / UI polling churn 점검
- 장기 실행 진단 포인트 추가

### D. 그룹 자동 합류 UX 고도화

목표:
- 그룹 구성원 중 한 명의 IP만 입력해도 새 PC가 그룹에 합류하도록 한다.

핵심 작업:
- 현재 그룹 참여 흐름 위에 초기화/충돌 안내 추가
- `nodes`, `layout`, monitor 관련 설정 동기화 범위 재정리
- 충돌 시 사용자 확인 흐름 설계

### E. 서비스 기반 로그인 전 제어

목표:
- 로그인 전 제어를 장기 과제로 검토한다.

핵심 작업:
- `Windows Service + 사용자 세션 UI + 안전한 입력 주입 계층` 구조 검토
- 일반 시작프로그램 기반 구조와 분리된 운영 모델 정리

### F. 실제 화면 크기 반영 레이아웃 보강

목표:
- 현재 논리 격자 배치 위에 실제 모니터 크기 차이와 빈 공간을 얼마나 반영할지 장기 과제로 검토한다.

핵심 작업:
- 레이아웃에서 노드 크기를 동일한 격자 칸이 아니라 실제 화면 크기 비율로 더 반영할지 검토
- 대형 스크린 / 소형 스크린이 섞인 환경에서 total width/height 기준 정렬 규칙 정리
- 중간 빈 공간이 있는 경우 edge routing과 gap jump를 어떤 정책으로 연결할지 정의
- 실제 사이즈 기반 배치와 현재 단순 격자 배치 사이의 호환 방식 검토

### G. 프로그램 최적화

목표:
- 다수 노드, 다중 모니터, 장시간 실행 기준에서도 UI와 입력 처리 비용이 과하게 늘어나지 않게 한다.

핵심 작업:
- redraw / rebuild / signal churn 측정 기반 점검
- 상태 polling 및 watcher 비용 점검
- layout/overview 갱신 비용 점검

완료 기준:
- 체감 끊김이나 불필요한 CPU churn이 눈에 띄게 줄어든다.

## 5. 범위 밖 / 보류 항목

현재 로드맵에서는 아래 항목을 바로 하지 않는다.

- 대규모 GUI 프레임워크 교체
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩/아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체
