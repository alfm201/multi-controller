# Multi Screen Pass Roadmap

## 1. 문서 목적

이 문서는 현재 코드베이스 기준으로 앞으로 다룰 엔지니어링 과제를 정리한다.

- 완료 보고나 과거 phase 기록은 길게 남기지 않는다.
- 우선순위는 `Now / Next / Later`로만 관리한다.
- 새 기능 추가보다 구조, 성능, 안정성, 운영성을 먼저 본다.

## 2. 현재 상태 요약

최근 작업으로 아래 항목은 한 단계 정리되었다.

- 입력/경계 처리 구조 분해
  - `display_state`, `edge_detection`, `edge_routing`, `edge_actions`, `routing_table`, `edge_runtime`로 책임 분리
- self 경계 라우팅 재설계
  - steady-state whole-display clip 제거
  - edge 단위 라우팅 구조로 전환
  - self 순방향/역방향 배치 실기기 확인 완료
- low-level hook 경로 안정화
  - Win32 hook 초기화 경로 정리
  - fallback 로그 가시성 개선

즉, 지금부터의 로드맵은 “경계 라우팅을 처음부터 다시 설계한다”가 아니라, 그 기반 위에서 남은 구조 작업과 운영 안정화를 이어가는 쪽에 가깝다.

## 3. Done

### A. 경계 라우팅 재설계

완료 내용:
- self/display 상태를 별도로 유지하도록 구조 정리
- raw hook 좌표는 방향 판단과 cross-axis ratio 계산 위주로 축소
- `GetCursorPos()`는 검증/복구 성격으로 격하
- steady-state whole-display clip 제거
- reverse 배치에서 중앙 block / 바깥 warp 동작 안정화
- blocked edge에서 한 축만 clamp되고 다른 축은 유지되도록 보정

완료 기준:
- self 순방향/역방향 배치에서 기대한 경계 동작이 실기기 기준으로 정상

### B. 경계 처리 모듈화 1차

완료 내용:
- `auto_switch`의 책임을 감지 / 라우팅 / 실행으로 분해
- layout 기반 edge candidate precompute 도입
- 관련 회귀 테스트 추가

완료 기준:
- 경계 동작 수정 시 영향을 받는 파일 범위가 예전보다 작고 읽기 쉬움

## 4. Now

### A. 프로젝트 구조 개선 및 모듈화

목표:
- UI 셸, 상태 계산, 레이아웃 편집, 설정 저장, Windows 상호작용의 경계를 더 분명하게 만든다.

핵심 작업:
- `status_window / status_view` 계열에서 화면 구성과 상태 계산을 더 분리
- `layout_editor / layout_dialogs / layouts` 계열에서 편집 모델과 Qt 위젯 책임을 더 줄이기
- `main.py` 조립 책임을 계속 축소

완료 기준:
- UI 수정과 런타임/OS 수정이 서로 덜 얽히고, 테스트도 같은 경계로 나뉜다.

### B. 프로그램 최적화

목표:
- 다수 노드, 다중 모니터, 장시간 실행 기준에서도 UI와 입력 처리가 부담 없이 유지되게 한다.

핵심 작업:
- 불필요한 redraw, rebuild, signal churn 줄이기
- 개요/레이아웃/모니터 맵 갱신 비용 줄이기
- 입력 큐와 상태 갱신 비용 관찰 지점 정리

완료 기준:
- 체감 끊김과 과한 CPU churn이 눈에 띄게 줄어든다.

## 5. Next

### A. Multi-node 실환경 검증 및 안정화

목표:
- self에서 검증된 경계 라우팅을 multi-node 환경까지 확장해서 확인한다.

핵심 작업:
- target active -> self 복귀 경계 동작 실기기 확인
- online/offline target 전환 동작 재검증
- 다른 노드 재접속 후 layout/bootstrap 동기화 재확인

완료 기준:
- self뿐 아니라 실제 target 전환까지 포함해 경계 동작을 설명 가능하고 재현 가능하다.

### B. UX 단순화와 정보 밀도 재정리

목표:
- 작은 창에서도 읽기 쉽고, 필요한 정보만 보이게 만든다.

핵심 작업:
- 상태 카드, 노드 목록, 설정 화면의 정보 과밀도 줄이기
- 중복 상태 표시와 긴 설명 문구를 계속 정리
- 작은 해상도에서도 무너지지 않는 기본 폭/여백 유지

완료 기준:
- 기본 창 크기가 작아도 핵심 흐름이 유지된다.

### C. 설정 저장 신뢰성 및 권한/예외 처리 점검

목표:
- GUI에서 설정 변경이 간헐적으로 반영되지 않는 문제를 구조적으로 검토한다.

핵심 작업:
- `LocalAppData` 기반 설정 저장 경로의 권한/잠금/예외 경로 점검
- 저장 실패 시 사용자 안내, 로그, 재시도 전략 검토
- backup prune와 설정 저장 타이밍이 충돌하지 않는지 확인

완료 기준:
- 설정 저장 실패가 조용히 묻히지 않고, 실패 원인과 사용자 경험이 명확해진다.

### D. 자동 업데이트 지원 검토

목표:
- GitHub Release 기반 수동 배포에서 자동 업데이트 지원 가능성을 검토한다.

핵심 작업:
- 버전 비교, 다운로드, 교체, 실패 복구 흐름 정리
- 설치 프로그램 기반 업데이트 UX 검토
- 코드 서명과 신뢰 UX까지 함께 고려

완료 기준:
- 자동 업데이트를 도입할지, 도입한다면 어떤 방식인지 판단 근거가 생긴다.

## 6. Later

### A. 입력 계층 재평가

목표:
- low-level hook 이후에도 더 낮은 계층이 필요한지 판단한다.

핵심 작업:
- hook 기반 구조의 한계와 운영 리스크 정리
- raw input 또는 더 낮은 입력 계층이 필요한 조건 문서화

### B. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포, 복구, 실행 경로, 백업 정책을 운영 문서와 함께 정리한다.

핵심 작업:
- installer/portable 기준 문서 정리
- recovery watchdog, 수동 복구 도구, backup 정책 정리
- 고정 listen 포트 충돌 시 예외 처리와 포트 풀 fallback 전략 검토

### C. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠 이상, 길게는 수개월 실행되는 환경을 전제로 병목을 줄인다.

핵심 작업:
- 입력 큐 적체 가능성 검토
- watcher / reporter / UI polling churn 점검
- 장기 운영 진단 포인트 추가

### D. 설정 마이그레이션 구조화

목표:
- legacy 설정 호환 코드를 중앙 마이그레이션 계층으로 정리한다.

핵심 작업:
- `config_version` 기반 마이그레이션 구조 검토
- 버전별 변환 함수를 한곳에 모으는 방식 설계

### E. 그룹 자동 합류 / 설정 동기화

목표:
- 그룹 구성원 한 명의 IP만 입력해도 새 PC가 그룹에 합류할 수 있게 한다.

핵심 작업:
- 현재 로컬 노드 설정 초기화 안내
- `nodes`, `layout`, monitor 관련 설정 동기화
- 충돌 시 사용자 확인 흐름 설계

### F. 서비스 기반 로그인 전 제어

목표:
- 로그인 전 상태 제어를 장기 과제로 따로 검토한다.

핵심 작업:
- `Windows Service + 사용자 세션 UI + 안전한 입력 주입 계층` 구조 검토
- 일반 시작프로그램 앱 구조와 분리된 운영 모델 정리

## 7. 범위 밖 / 보류 항목

현재 로드맵에서는 아래 항목을 바로 다루지 않는다.

- 대규모 GUI 프레임워크 교체
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩/아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체
- 로그인 전 제어를 단기 과제로 끌어오는 것

## 8. 바로 다음 실행 후보

1. multi-node 실환경 검증
2. 설정 저장 신뢰성 점검
3. UI/상태 화면 모듈화 2차
