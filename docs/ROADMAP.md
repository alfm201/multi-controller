# Multi Screen Pass Roadmap

## 1. 문서 목적

이 문서는 현재 코드베이스 기준으로 앞으로 다룰 엔지니어링 과제를 정리한다.

- 완료 보고나 과거 phase 기록은 최소화한다.
- 우선순위는 `Now / Next / Later`로만 관리한다.
- 새 기능 추가보다 구조, 안정성, 성능, 운영 경험을 먼저 다룬다.

## 2. 현재 상태 요약

최근 작업으로 아래 항목은 한 단계 정리됐다.

- 입력/경계 처리 구조 분해
  - `display_state`, `edge_detection`, `edge_routing`, `edge_actions`, `routing_table`, `edge_runtime`로 책임 분리
- self 경계 라우팅 재설계
  - steady-state whole-display clip 제거
  - edge 단위 라우팅 구조로 전환
  - self 순방향/역방향 배치 기준 동작 검증 완료
- low-level hook 안정화
  - Win32 hook 초기화 경로 정리
  - fallback 경로 가시성 보강
- 설정 저장 신뢰성 1차 보강
  - 저장 직렬화
  - atomic write retry
  - layout/settings 통합 저장
  - background save 실패 가시화

즉 현재 로드맵은 경계 처리의 재설계가 끝난 뒤, multi-node 검증과 운영 안정성을 더 다듬는 단계에 가깝다.

## 3. Done

### A. 경계 라우팅 재설계

완료 내용:
- self/display 상태를 별도 상태로 관리
- raw move 기반 방향 판단과 edge 판정 분리
- `GetCursorPos()`는 검증/보조 역할로 격하
- reverse 배치에서 중앙 block / 바깥 warp 동작 정리
- blocked edge에서 축 보존 동작 정리

완료 기준:
- self 단일/듀얼 모니터 기준 경계 라우팅이 테스트와 실기기에서 일관되게 동작한다.

### B. 입력/경계 처리 모듈화 1차

완료 내용:
- `auto_switch`의 감지 / 라우팅 / 실행 책임 분리
- layout 기반 edge candidate precompute 도입
- 회귀 테스트 추가

완료 기준:
- 경계 동작 수정 시 영향 범위가 이전보다 좁고, 테스트도 같은 경계로 나뉜다.

### C. 설정 저장 신뢰성 1차 보강

완료 내용:
- config 저장 경로 직렬화
- `tmp -> replace` 저장에 대한 retry/backoff 추가
- 고정 tmp 파일명 제거
- layout + settings 통합 저장 경로 추가
- background save 실패 시 UI 메시지 노출 경로 추가

완료 기준:
- 짧은 파일 잠금이나 저장 타이밍 충돌로 인한 간헐 저장 실패 가능성을 줄이고, 실패 시 사용자와 로그에 더 잘 드러난다.

## 4. Now

### A. 프로젝트 구조 개선 및 모듈화 2차

목표:
- UI 셸, 상태 계산, 설정 저장, Windows 상호작용 경계를 더 분명하게 만든다.

핵심 작업:
- `status_window / status_view` 계열에서 화면 구성과 상태 계산 책임 분리
- `layout_editor / layout_dialogs / layouts` 계열에서 편집 모델과 Qt 이벤트 책임 축소
- `main.py` 조립 책임을 더 줄이고 초기화 흐름을 정리

완료 기준:
- UI 수정과 OS 연동 수정이 서로 다른 파일/테스트 경계 안에서 끝난다.

### B. 프로그램 최적화

목표:
- 다수 노드, 다중 모니터, 장시간 실행 기준에서도 UI와 입력 처리 비용이 과하게 늘어나지 않게 한다.

핵심 작업:
- redraw / rebuild / signal churn 점검
- 상태 polling 및 watcher 비용 점검
- layout/overview 갱신 비용 점검

완료 기준:
- 체감 끊김이나 불필요한 CPU churn이 눈에 띄게 줄어든다.

## 5. Next

### A. Multi-node 실환경 검증 및 안정화

목표:
- self에서 검증한 경계 라우팅을 실제 multi-node 환경까지 확장해 확인한다.

핵심 작업:
- target active -> self 복귀 경계 동작 확인
- online/offline target 전환 흐름 확인
- rejoin 후 layout/bootstrap 동기화 재검증
- online 상태인데 layout revision/hash가 어긋나는 경우 감지
- revision mismatch 시 coordinator snapshot 재요청 또는 강제 재동기화 경로 검토

완료 기준:
- self만이 아니라 실제 target 전환까지 포함한 경계 동작을 재현하고 설명할 수 있다.
- online 노드 간 layout divergence가 발생했을 때 기준(coordinator)과 복구 경로를 설명할 수 있다.

### B. UI/상태 화면 단순화 2차

목표:
- 작은 창에서도 읽기 쉽고, 필요한 정보만 보이게 만든다.

핵심 작업:
- 상태 카드, 노드 목록, 설정 화면 정보 밀도 재조정
- 중복 상태 표시와 긴 안내 문구 축소

완료 기준:
- 기본 창 크기가 작아도 주요 흐름을 읽고 조작하는 데 무리가 없다.

### C. 설정 저장 신뢰성 2차 점검

목표:
- 1차 보강 이후에도 실제 사용자 환경에서 남는 저장 실패 패턴을 줄인다.

핵심 작업:
- 장시간 사용, 백신/동기화 도구, 편집기 동시 열림 환경에서 재현 확인
- 부분 저장 상태나 장시간 파일 점유 시 복구/안내 방식 점검
- 설정 저장 실패 메시지 톤과 안내 개선

완료 기준:
- “다른 프로세스가 파일을 사용 중입니다”, 접근 권한, tmp 교체 실패 류 이슈가 실제 사용 환경에서 현저히 줄고, 남더라도 원인 파악이 가능하다.

### D. 자동 업데이트 지원 검토

목표:
- GitHub Release 기반 수동 배포에서 자동 업데이트로 갈지 판단한다.

핵심 작업:
- 버전 비교, 다운로드, 교체, 실패 복구 흐름 정리
- installer 기반 업데이트 UX 검토
- 코드 서명과 신뢰 UX 고려

완료 기준:
- 자동 업데이트를 도입할지, 도입한다면 어떤 방식일지 결정 근거가 정리된다.

## 6. Later

### A. 입력 계층 재평가

목표:
- low-level hook 이후에도 더 낮은 입력 계층이 필요한지 판단한다.

핵심 작업:
- hook 기반 구조의 한계와 운영 리스크 정리
- raw input 또는 더 낮은 계층이 필요한 조건 문서화

### B. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포, 복구, 실행 경로, 백업 정책을 운영 문서와 함께 정리한다.

핵심 작업:
- installer/portable 기준 문서화
- recovery watchdog / 수동 복구 / backup 정책 정리
- 포트 충돌 시 예외 처리 및 포트 풀 fallback 검토

### C. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠 이상 장시간 실행하는 환경에서 병목과 누적 문제를 줄인다.

핵심 작업:
- 입력 큐 적체 가능성 점검
- watcher / reporter / UI polling churn 점검
- 장기 실행 진단 포인트 추가

### D. 설정 마이그레이션 구조화

목표:
- legacy 설정 호환 코드를 중앙 마이그레이션 계층으로 정리한다.

핵심 작업:
- `config_version` 기반 마이그레이션 구조 검토
- 버전별 변환 함수를 중앙에서 관리하는 방식 설계

### E. 그룹 자동 합류 / 설정 동기화

목표:
- 그룹 구성원 중 한 명의 IP만 입력해도 새 PC가 그룹에 합류하도록 한다.

핵심 작업:
- 현재 로컬 노드 설정 초기화 안내
- `nodes`, `layout`, monitor 관련 설정 동기화
- 충돌 시 사용자 확인 흐름 설계

### F. 서비스 기반 로그인 전 제어

목표:
- 로그인 전 제어를 장기 과제로 검토한다.

핵심 작업:
- `Windows Service + 사용자 세션 UI + 안전한 입력 주입 계층` 구조 검토
- 일반 시작프로그램 기반 구조와 분리된 운영 모델 정리

### G. 실제 화면 크기 반영 레이아웃 모델

목표:
- gap jump 여부뿐 아니라, 실제 모니터 크기 차이와 빈 공간까지 반영한 물리 배치 모델을 장기 과제로 검토한다.

핵심 작업:
- 레이아웃에서 노드 크기를 동일한 격자 칸이 아니라 실제 화면 크기 비율로 표현할지 검토
- 대형 스크린 / 소형 스크린이 섞인 환경에서 total width/height 기준 정렬 규칙 정리
- 중간 빈 공간이 있는 경우 edge routing과 gap jump를 어떤 정책으로 연결할지 정의
- 실제 사이즈 기반 배치와 현재 단순 격자 배치 사이의 호환 방식 검토

## 7. 범위 밖 / 보류 항목

현재 로드맵에서는 아래 항목을 바로 하지 않는다.

- 대규모 GUI 프레임워크 교체
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩/아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체

## 8. 바로 다음 실행 후보

1. multi-node 실환경 검증
2. 설정 저장 신뢰성 2차 점검
3. UI/상태 화면 모듈화 2차
