# Multi Screen Pass Roadmap

## 1. 문서 목적

이 문서는 현재 코드베이스 기준으로 앞으로 다룰 엔지니어링 과제를 정리한다.

- 완료 보고나 과거 phase 기록은 최소화한다.
- 우선순위는 `Now / Next / Later`로만 관리한다.
- 새 기능 추가보다 구조, 안정성, 성능, 운영 경험을 먼저 다룬다.

## 2. Now

### A. Multi-node 실환경 검증 및 안정화

목표:
- self에서 검증한 경계 라우팅을 실제 multi-node 환경까지 확장해 확인한다.

핵심 작업:
- target active -> self 복귀 경계 동작 확인
- online/offline target 전환 흐름 확인
- rejoin 후 layout/bootstrap 동기화 재검증
- online 상태인데 layout revision/hash가 어긋나는 경우 감지
- revision mismatch 시 coordinator snapshot 재요청 또는 강제 재동기화 경로 정리

완료 기준:
- self만이 아니라 실제 target 전환까지 포함한 경계 동작을 재현하고 설명할 수 있다.
- online 노드 간 layout divergence가 발생했을 때 기준(coordinator)과 복구 경로를 설명할 수 있다.

### B. 노드 식별자 분리

목표:
- 현재 `config.nodes[].name`을 내부 식별자처럼 사용하는 구조를 분리하고, 표시명과 고정 식별자를 따로 관리한다.

핵심 작업:
- 설치/초기 설정 시 생성되는 immutable `node_id` 도입 검토
- `name`은 표시명 기본값으로만 사용하고, hostname은 self auto-detect 보조값으로만 축소
- coordinator, layout, monitor inventory, node sync 경로가 `name` 대신 `node_id`를 기준으로 동작하도록 마이그레이션 설계
- 기존 config와 이미 저장된 layout/monitor 관련 데이터의 마이그레이션 전략 정리
- VM/복제 환경, hostname 변경, 표시명 변경 시 동일 노드로 유지되는지 검증

완료 기준:
- 표시명 변경이나 hostname 변경이 내부 식별자 충돌 없이 동작하고, 기존 설정도 안전하게 올릴 수 있다.

### C. 런타임 조립 경계 잔여 정리

목표:
- 이미 분리된 UI/레이아웃/설정 모듈 위에 남아 있는 조립 책임만 정리해, 수정 범위를 더 예측 가능하게 만든다.

핵심 작업:
- `main.py` 초기화/배선 책임 추가 축소
- 흩어진 상수값과 기본 설정값을 한곳에서 관리하는 상수 모듈 분리 검토
- 로그 메시지 태그(`[CONFIG]`, `[AUTO]`, `[INPUT]` 등) 규칙을 짧고 일관된 도메인 기준으로 정리 검토

완료 기준:
- 새 기능 없이도 시작 경로, 기본값, 로그 규칙을 한 번에 파악할 수 있고, UI 수정과 런타임 배선 수정의 경계가 더 분명해진다.

## 3. Next

### A. 설정 저장 신뢰성 2차 점검

목표:
- 1차 보강 이후에도 실제 사용자 환경에서 남는 저장 실패 패턴을 줄인다.

핵심 작업:
- 장시간 사용, 백신/동기화 도구, 편집기 동시 열림 환경에서 재현 확인
- 부분 저장 상태나 장시간 파일 점유 시 복구/안내 방식 점검
- 설정 저장 실패 메시지 톤과 안내 개선

완료 기준:
- “다른 프로세스가 파일을 사용 중입니다”, 접근 권한, tmp 교체 실패 류 이슈가 실제 사용 환경에서 현저히 줄고, 남더라도 원인 파악이 가능하다.

### B. 설정 마이그레이션 체계 도입

목표:
- version 없는 split config 저장 구조를 중앙 마이그레이션 계층으로 끌어올린다.

핵심 작업:
- `config_version` 기반 마이그레이션 구조 검토
- 버전별 변환 함수를 중앙에서 관리하는 방식 설계
- `node_id` 분리와 저장 포맷 변경이 생겨도 업그레이드 경로가 단일 지점에서 관리되도록 정리

완료 기준:
- 새 설정 구조를 추가할 때 ad-hoc 호환 코드를 여기저기 흩뿌리지 않고, 명시적인 버전 업그레이드 경로로 처리할 수 있다.

### C. 자동 업데이트 정책 정리

목표:
- 이미 있는 업데이트 기능을 어떤 배포 정책과 UX로 운영할지 결정한다.

핵심 작업:
- 수동 확인 / 자동 확인 / 자동 설치 범위를 구분
- installer 기반 업데이트 UX와 실패 복구 흐름 점검
- 코드 서명과 신뢰 UX를 포함한 배포 정책 정리

완료 기준:
- 자동 업데이트를 기본 정책으로 둘지, 수동 설치 중심으로 유지할지와 그 근거가 정리된다.

### D. UI/상태 화면 단순화 2차

목표:
- 작은 창에서도 읽기 쉽고, 필요한 정보만 보이게 만든다.

핵심 작업:
- 상태 카드, 노드 목록, 설정 화면 정보 밀도 재조정
- 중복 상태 표시와 긴 안내 문구 축소

완료 기준:
- 기본 창 크기가 작아도 주요 흐름을 읽고 조작하는 데 무리가 없다.

## 4. Later

### A. 입력 계층 재평가

목표:
- low-level hook 이후에도 더 낮은 입력 계층이 필요한지 판단한다.

핵심 작업:
- hook 기반 구조의 한계와 운영 리스크 정리
- raw input 또는 더 낮은 계층이 필요한 조건 문서화

### B. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포, 복구, 실행 경로, 백업 정책을 운영 문서와 함께 정리한다.

핵심 작업:
- installer/portable 기준 문서화
- recovery watchdog / 수동 복구 / backup 정책 정리
- 포트 충돌 시 예외 처리 및 포트 풀 fallback 검토

### C. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠 이상 장시간 실행하는 환경에서 병목과 누적 문제를 줄인다.

핵심 작업:
- 입력 큐 적체 가능성 점검
- watcher / reporter / UI polling churn 점검
- 장기 실행 진단 포인트 추가

### D. 그룹 자동 합류 UX 고도화

목표:
- 그룹 구성원 중 한 명의 IP만 입력해도 새 PC가 그룹에 합류하도록 한다.

핵심 작업:
- 현재 그룹 참여 흐름 위에 초기화/충돌 안내 추가
- `nodes`, `layout`, monitor 관련 설정 동기화 범위 재정리
- 충돌 시 사용자 확인 흐름 설계

### E. 서비스 기반 로그인 전 제어

목표:
- 로그인 전 제어를 장기 과제로 검토한다.

핵심 작업:
- `Windows Service + 사용자 세션 UI + 안전한 입력 주입 계층` 구조 검토
- 일반 시작프로그램 기반 구조와 분리된 운영 모델 정리

### F. 실제 화면 크기 반영 레이아웃 보강

목표:
- 현재 논리 격자 배치 위에 실제 모니터 크기 차이와 빈 공간을 얼마나 반영할지 장기 과제로 검토한다.

핵심 작업:
- 레이아웃에서 노드 크기를 동일한 격자 칸이 아니라 실제 화면 크기 비율로 더 반영할지 검토
- 대형 스크린 / 소형 스크린이 섞인 환경에서 total width/height 기준 정렬 규칙 정리
- 중간 빈 공간이 있는 경우 edge routing과 gap jump를 어떤 정책으로 연결할지 정의
- 실제 사이즈 기반 배치와 현재 단순 격자 배치 사이의 호환 방식 검토

### G. 프로그램 최적화

목표:
- 다수 노드, 다중 모니터, 장시간 실행 기준에서도 UI와 입력 처리 비용이 과하게 늘어나지 않게 한다.

핵심 작업:
- redraw / rebuild / signal churn 측정 기반 점검
- 상태 polling 및 watcher 비용 점검
- layout/overview 갱신 비용 점검

완료 기준:
- 체감 끊김이나 불필요한 CPU churn이 눈에 띄게 줄어든다.

## 5. 범위 밖 / 보류 항목

현재 로드맵에서는 아래 항목을 바로 하지 않는다.

- 대규모 GUI 프레임워크 교체
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩/아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체

2. 노드 식별자 분리 설계
