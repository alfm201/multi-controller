# Multi Screen Pass Roadmap

## 1. 문서 목적

이 문서는 현재 코드베이스를 기준으로 앞으로 손봐야 할 엔지니어링 과제를 정리한다.

- 완료 보고나 과거 phase 기록은 적지 않는다.
- 우선순위는 `Now / Next / Later`로만 관리한다.
- 기능 추가보다 구조, 성능, 안정성, 운영 편의성을 먼저 본다.

## 2. 현재 구조 진단

현재 앱은 기능적으로는 많이 올라왔지만, 아래 영역에 책임이 다소 몰려 있다.

- `status_window / status_view`
  - 화면 셸, 상태 가공, 카드/표 렌더링이 강하게 엮여 있다.
- `layout_editor / layout_dialogs / layouts`
  - 레이아웃 모델, 캔버스 표시, 모니터 맵 편집, 좌표 해석이 한 덩어리로 움직인다.
- `input_capture / auto_switch / local_cursor / clip_recovery`
  - 입력 캡처, 경계 처리, 커서 clip, 복구 도구가 같은 문제 공간을 공유한다.
- `main.py`
  - 네트워크, GUI, 입력, 진단, 백업 정리, 복구 초기화까지 조립 책임이 넓다.

현재 가장 큰 개선 축은 두 가지다.

1. 구조 개선 및 모듈화  
2. 프로그램 최적화와 장기 실행 안정성

## 3. Now / Next / Later

### Now

#### A. 프로젝트 구조 개선 및 모듈화

목표:
- 변경 영향 범위를 줄이고, 화면 로직과 OS/입력 로직의 경계를 분명히 한다.

핵심 작업:
- Qt UI 셸, 상태 계산, 레이아웃 편집, 모니터 맵 편집, Windows 상호작용, 설정/저장 로직의 책임을 다시 자른다.
- 큰 runtime 모듈을 `화면 셸 / 상태 모델 / 편집 도메인 / OS 연동` 기준으로 정리한다.
- 테스트도 같은 경계에 맞춰 재배치한다.

완료 기준:
- UI 수정 시 함께 건드리는 파일 범위가 지금보다 줄어든다.
- 화면 수정과 입력/OS 수정이 서로 덜 얽힌다.

#### B. 프로그램 최적화

목표:
- 다수 노드, 다중 모니터, 장시간 실행에서도 UI와 입력 흐름이 가볍게 유지되게 한다.

핵심 작업:
- 상태 갱신, 테이블 렌더링, 레이아웃 캔버스, 모니터 맵 드래그에서 불필요한 rebuild를 줄인다.
- redraw 횟수, 불필요한 signal, 동기 작업, 과한 polling을 줄인다.
- 대량 노드 테스트에서도 기본 창 사용감이 무너지지 않게 한다.

완료 기준:
- 개요, 레이아웃, 모니터 맵에서 체감 끊김이 눈에 띄게 줄어든다.
- 장시간 실행 시 과한 CPU churn이나 UI 갱신 떨림이 줄어든다.

### Next

#### C. 입력/경계 처리 신뢰성 강화

목표:
- self 내부 모니터 이동, dead edge, offline edge, clip lifecycle을 설명 가능하고 재현 가능한 상태로 고정한다.

핵심 작업:
- self-display warp, dead edge, offline edge, clip recovery를 상태기계 관점에서 다시 정리한다.
- startup recovery, watchdog, 수동 복구 툴, debug logging의 역할을 정리한다.
- 경계 처리 로그는 원인 파악에는 충분하고 운영 시에는 과하지 않게 정리한다.

완료 기준:
- 경계 관련 이슈를 재현, 설명, 복구하는 흐름이 문서와 코드에서 일치한다.

#### D. UX 단순화와 정보 밀도 재정리

목표:
- 화면에 보이는 정보와 액션을 덜어내고, 작은 창에서도 읽기 쉬운 상태를 만든다.

핵심 작업:
- 개요, 레이아웃, 설정, 노드 관리에서 중복 상태표시와 과한 설명 텍스트를 줄인다.
- 탭별 역할을 더 분명히 하고, 작은 화면에서도 무너지지 않는 기본 배치를 만든다.
- 운영에 꼭 필요한 정보만 남기고 나머지는 hover, dialog, 고급 정보로 숨긴다.

완료 기준:
- 기본 창 크기가 더 작아져도 핵심 정보와 조작 흐름이 유지된다.

#### E. 그룹 자동 참가 / 설정 동기화

목표:
- 새 PC가 그룹에 합류할 때 수동으로 `nodes`, `layout`, monitor 설정을 복사하지 않아도 되게 한다.

핵심 작업:
- 그룹 구성원 중 한 PC의 IP만 입력하면 연결 요청을 보내고, 현재 로컬의 노드 관련 설정을 초기화한 뒤 그룹 설정을 동기화해 받아오는 흐름을 설계한다.
- 이 작업은 “현재 연결 설정을 초기화합니다” 같은 명확한 안내와 확인 단계를 포함한다.
- 동기화 대상은 최소한 `nodes`, `layout`, monitor 관련 설정을 포함하고, 충돌 시 로컬 덮어쓰기 규칙을 문서화한다.

완료 기준:
- 새 PC를 그룹에 넣는 과정이 수동 JSON 편집 없이 진행 가능하다.
- 사용자는 한 번의 확인으로 그룹 설정을 받아와 실행할 수 있다.

### Later

#### F. 입력 계층 재평가

목표:
- 필요하다면 Windows 전용 저수준 입력 처리로 내려갈지 판단 근거를 만든다.

핵심 작업:
- 현재 `pynput + OS 보정` 구조의 한계와 장점을 문서화한다.
- low-level hook 또는 raw input 기반 설계가 실제로 필요한지 검토한다.

완료 기준:
- 입력 계층을 유지할지 교체할지 근거를 가지고 결정할 수 있다.

#### G. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포, 설정 초기화, 백업, 복구 도구 사용 경험을 더 단순하게 만든다.

핵심 작업:
- 배포 폴더 구조, 설정 파일 초기화/검증/마이그레이션, 복구 도구, 백업 정책을 사용자 문서와 운영 문서로 정리한다.
- 실행 파일 기준 설정 경로와 백업/복구 흐름을 더 명확히 만든다.
- 고정 listen 포트 사용을 전제로, 포트 충돌이나 시작 실패 시 예외 처리와 사용자 메시지 흐름을 정리한다.
- 필요하다면 고정 포트 단일값 대신 제한된 포트 풀 범위를 fallback으로 쓰는 방식도 함께 검토한다.

완료 기준:
- 배포와 복구 절차를 문서만 보고 따라갈 수 있다.

#### H. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠에서 몇 달 이상 켜 둘 수 있는 운영 기준에서 입력, UI, 복구 루프가 안정적으로 유지되게 한다.

핵심 작업:
- 입력 큐 적체 가능성과 mouse move coalescing 필요성을 검토한다.
- watcher, reporter, GUI diff 갱신이 장시간 실행에서 만드는 CPU / 메모리 churn을 줄인다.
- 큐 길이, 갱신 비용, 복구 루프 상태를 추적할 수 있는 진단 포인트를 추가한다.

완료 기준:
- 장기 실행 시 병목과 리스크를 설명할 수 있고, 운영 중 관찰 가능한 지표가 있다.

## 4. 바로 다음 실행 후보 3개

### 1) Runtime 모듈 분해 1차

목표:
- `status_window`, `status_view`, `layout_editor`, `layout_dialogs`, `layouts`의 책임을 더 명확히 나눈다.

산출물:
- 화면 셸 / 상태 모델 / 편집 로직 / Windows 연동 경계 정리
- 관련 테스트 재배치

완료 기준:
- UI 수정 시 수정 파일 범위가 줄고, 테스트도 같은 경계로 나뉜다.

### 2) 렌더링 / 입력 성능 패스

목표:
- 개요 표, 레이아웃 캔버스, 모니터 맵, 상태 polling 비용을 줄인다.

산출물:
- 불필요한 rebuild 제거
- selection / render sync 정리
- 대량 노드에서도 usable한 기본 상태

완료 기준:
- 대량 노드 + 레이아웃 이동/편집 시 끊김이 눈에 띄게 줄어든다.

### 3) 입력 경계 / 커서 안전성 정리

목표:
- self-display warp, dead edge, clip recovery를 더 안정적으로 고정한다.

산출물:
- 경계 처리 상태기계 문서화
- debug 로그 정리
- 복구 흐름 명확화

완료 기준:
- 현재 남아 있는 경계 처리 이슈를 재현, 설명, 복구하는 흐름이 정리된다.

## 5. 범위 밖 / 보류 항목

현재 로드맵에는 아래 항목을 넣지 않는다.

- 대규모 GUI 프레임워크 재교체
- 기능 위주의 새 대형 feature 추가
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩/아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체
