# multi-controller Roadmap

## 1. 문서 목적

이 문서는 앞으로의 엔지니어링 방향을 정리하는 내부 기준 문서다.

- 완료 보고, 차수별 회고, 과거 phase 표현은 남기지 않는다.
- 현재 코드베이스를 기준으로 `Now / Next / Later` 우선순위만 관리한다.
- 기능 나열보다 구조, 성능, 안정성, 운영성 개선 순서를 우선한다.

## 2. 현재 구조 진단

현재 프로젝트는 기능적으로는 많이 올라왔지만, 구조적으로는 다음 덩어리에 부담이 몰려 있다.

- `status_window / status_view` 계열:
  화면 셸, 상태 해석, 선택 동기화, 요약 카드, 표 렌더링이 강하게 연결돼 있다.
- `layout_editor / layout_dialogs / layouts` 계열:
  레이아웃 편집, 모니터 맵 편집, 레이아웃 모델, 좌표 해석, 뷰 갱신 책임이 한 흐름에 섞여 있다.
- 입력 및 커서 처리 계열:
  `input_capture / auto_switch / local_cursor / clip_recovery`가 실제 OS 입력과 강하게 엮여 있어, 경계 처리와 복구 로직을 다룰 때 영향 범위가 크다.
- 런타임 조립 계층:
  `main.py`가 네트워크, GUI, 입력 캡처, 설정 복구, watchdog까지 넓게 조립하고 있어 진입 흐름이 무거운 편이다.

현재 병목은 크게 두 축이다.

1. 구조적으로 큰 모듈에 책임이 많이 몰려 있어 수정 범위가 넓다.  
2. UI 갱신, 레이아웃 렌더링, 입력/경계 처리에서 체감 성능과 디버깅 난도가 함께 올라가고 있다.

추가로 이 프로젝트는 한 번 실행하면 장기간 계속 켜 두는 운영 형태를 전제로 보아야 한다.

- 입력 캡처 큐가 무한정 커질 수 있는지
- 상태 polling과 주기 작업이 장시간 실행에서 CPU / 메모리 churn을 만드는지
- watcher, reporter, GUI 갱신, 복구 스레드가 몇 주~몇 달 단위로 안정적으로 유지되는지

같은 항목은 별도의 장기 실행 안정성 관점에서 점검이 필요하다.

## 3. Now / Next / Later

### Now

#### A. 프로젝트 구조 개선 및 모듈화

목표:
- 변경 영향 범위를 줄이고, 각 모듈의 책임을 더 분명하게 나눈다.

핵심 작업:
- Qt UI 셸, 상태 계산, 레이아웃 편집, 모니터 맵 편집, Windows 상호작용, 설정 저장 로직의 경계를 다시 자른다.
- 큰 runtime 모듈을 `화면 셸 / 상태 모델 / 편집 도메인 / OS 연동` 기준으로 재정리한다.
- 테스트도 같은 경계에 맞춰 재배치해, 화면 수정과 OS 입력 수정이 서로 덜 얽히게 만든다.

완료 기준:
- 한 화면 또는 한 입력 흐름을 수정할 때 건드리는 파일 범위가 지금보다 눈에 띄게 줄어든다.
- 테스트가 UI, 상태 해석, OS 상호작용 경계에 맞게 읽힌다.

#### B. 프로그램 최적화

목표:
- 대량 노드, 다중 모니터, 장시간 실행 기준에서도 UI와 입력 흐름이 가볍게 유지되게 한다.

핵심 작업:
- 상태 갱신, 표 렌더링, 레이아웃 캔버스, 모니터 맵 드래그, 입력 경계 처리의 비용을 줄인다.
- 전체 rebuild보다 diff 적용, 불필요한 signal 감소, 동기 작업 축소를 우선한다.
- “느낌상 느리다”가 아니라 redraw 횟수, selection sync, polling 비용을 줄이는 방식으로 접근한다.

완료 기준:
- 개요 표, 레이아웃 이동, 모니터 맵 편집에서 끊김과 깜빡임이 줄어든다.
- 다수 노드 테스트에서도 기본 창 사용감이 무너지지 않는다.

### Next

#### C. 입력/경계 처리 신뢰성 강화

목표:
- self 내부 모니터 이동, dead edge, offline edge, cursor clip 복구 흐름을 더 설명 가능하고 재현 가능하게 만든다.

핵심 작업:
- self-display warp, dead edge, offline edge, clip lifecycle을 상태기계 관점으로 다시 정리한다.
- watchdog, startup recovery, 수동 복구 도구, debug logging의 역할을 명확히 나눈다.
- 현재 남아 있는 “동작은 하지만 설명이 어려운” 예외 흐름을 줄인다.

완료 기준:
- 경계 관련 문제를 재현, 로그 해석, 복구까지 한 흐름으로 설명할 수 있다.
- 디버그 로그가 원인 파악에는 충분하지만 과하게 noisy하지 않다.

#### D. UX 단순화와 화면 정보 밀도 재정리

목표:
- 정보는 더 적게, 행동은 더 분명하게 보여주는 방향으로 화면을 다시 다듬는다.

핵심 작업:
- 개요, 레이아웃, 설정, 노드 관리에서 중복 상태 표시와 긴 설명 문구를 줄인다.
- 작은 창 폭에서도 무너지지 않는 기본 레이아웃을 유지한다.
- 상태 표시, 노드 선택, 설정 편집의 역할을 더 명확히 나눈다.

완료 기준:
- 기본 창 크기와 좁은 폭에서도 핵심 정보가 남는다.
- 새 사용자가 봐도 “어디서 보고, 어디서 수정하는지”가 더 즉시 읽힌다.

### Later

#### E. 입력 계층 재평가

목표:
- 필요하다면 Windows 전용 저수준 입력 처리로 내려갈지 판단할 수 있는 근거를 만든다.

핵심 작업:
- 현재 `pynput + OS 보정` 구조의 한계와 장점을 문서화한다.
- low-level hook 또는 raw input 기반 재설계가 정말 필요한지 판단한다.

완료 기준:
- 입력 계층을 유지할지, 더 낮은 레벨로 옮길지 근거를 갖고 결정할 수 있다.

#### F. 패키징 / 운영 / 복구 경험 정리

목표:
- 배포와 운영 복구 흐름을 사용자 입장에서 더 단순하게 만든다.

핵심 작업:
- 설정 초기화, migration, validate, 백업, 복구 도구, 수동 복구 exe 문서를 정리한다.
- 배포 시 필요한 파일 구조와 운영 가이드를 문서 기준으로 정리한다.

완료 기준:
- 새 환경 배포와 복구 절차를 문서만 보고 따라갈 수 있다.

#### G. 장기 실행 안정성 / 메모리 운영 하드닝

목표:
- 며칠이 아니라 몇 주~몇 달 단위로 켜 둬도 입력, UI, 복구 루프가 안정적으로 유지되게 만든다.

핵심 작업:
- 입력 캡처 큐의 무한 적체 가능성을 점검하고, 필요한 경우 bounded queue 또는 mouse move coalescing을 도입한다.
- 상태 polling, watcher, reporter, GUI diff 갱신의 주기 작업이 장시간 실행에서 만드는 CPU / 메모리 churn을 줄인다.
- 장기 실행 중 큐 길이, 상태 갱신 비용, 복구 루프, 스레드 수를 추적할 수 있는 운영 진단 포인트를 추가한다.

완료 기준:
- 장시간 실행 중 메모리 사용량과 큐 적체가 통제 가능한 범위에 머문다.
- 장기 실행 리스크를 재현, 측정, 설명할 수 있는 진단 기준이 마련된다.

## 4. 바로 다음 실행 후보 3개

### 1) Runtime 모듈 분해 1차

목표:
- `status_window`, `status_view`, `layout_editor`, `layout_dialogs`, `layouts` 책임을 재정리한다.

산출물:
- 화면 셸 / 상태 모델 / 편집 로직 / Windows 연동 경계 정리
- 큰 모듈의 1차 분리와 테스트 경계 재배치

완료 기준:
- UI 수정 시 수정 파일 범위가 줄고, 테스트도 같은 경계로 읽힌다.

### 2) 렌더링 / 입력 성능 패스

목표:
- 개요 표, 레이아웃 캔버스, 모니터 맵, 상태 polling 비용을 줄인다.

산출물:
- 불필요한 rebuild 제거
- selection / render sync 정리
- 다량 노드 환경에서도 버티는 기본 UI 상태

완료 기준:
- 다량 노드 + 레이아웃 이동/편집 시 끊김이 눈에 띄게 감소한다.

### 3) 입력 경계 / 커서 안전성 정리

목표:
- self-display warp, dead edge, clip recovery를 더 안정적으로 고정한다.

산출물:
- 경계 처리 상태기계 문서화
- debug 로그 축소 및 정리
- 복구 흐름과 watchdog 역할 정리

완료 기준:
- 현재 남아 있는 경계 관련 혼란을 재현, 설명, 복구 관점에서 정리할 수 있다.

## 5. 범위 밖 / 보류 항목

현재 로드맵에는 아래 항목을 넣지 않는다.

- 대규모 GUI 프레임워크 재교체
- 기능 위주의 새 대형 feature 추가
- OS 디스플레이 설정 자체를 직접 제어하는 기능
- 브랜딩 / 아이콘 전체 리디자인
- 네트워크 아키텍처 전면 교체
