# multi-controller Roadmap

## 1. 문서 목적

이 문서는 `multi-controller`의 앞으로 남은 큰 방향만 정리한다.

- 완료 기록이나 차수 회고는 적지 않는다.
- 세부 구현 절차는 `agent/PLAN.md`에서 관리한다.
- 이 문서는 지금 기준의 `Now / Next / Later` 우선순위만 남긴다.

## 2. 제품 방향

- 여러 PC를 하나의 자연스러운 작업 공간처럼 느끼게 만든다.
- GUI는 디버그 패널이 아니라 실사용 제어 패널이어야 한다.
- 모니터 맵은 계속 `실제 감지된 모니터`를 기준으로 동작한다.
- 설정은 `노드 기본 정보 / 레이아웃 / 모니터 override / 감지 캐시`처럼 책임이 분리된 상태를 유지한다.
- 운영 작업은 가능하면 GUI 안에서 끝나야 한다.

## 3. 현재 핵심 관심사

1. 실제 운용 중 모니터가 바뀌었을 때, 현재 감지 결과와 저장된 물리 override 차이를 더 분명하게 보여줄 필요가 있다.
2. 감지 캐시가 오래되었거나 특정 노드가 오프라인일 때, 사용자가 “현재 정보가 최신인지”를 즉시 알기 어렵다.
3. 노드 GUI 관리가 들어갔지만, 이름 변경이나 자기 자신 설정 변경처럼 재시작이 필요한 경로는 아직 보수적으로 막혀 있다.
4. 분리된 설정 파일 구조는 도입됐지만, 예제/도구/운영 문서 차원에서 더 쉽게 이해되게 다듬을 여지가 있다.

## 4. 장기 로드맵

### Track A. 모니터 변경 diff와 freshness 개선 (`Now`)

목표:
- 감지 결과가 바뀌었을 때 무엇이 달라졌는지, 그리고 지금 정보가 최신인지 더 분명하게 보여준다.

핵심 방향:
- 노드별 마지막 감지 시각과 stale 여부를 overview / 연결 상태 / 모니터 맵에 드러낸다.
- 감지된 logical 배치와 저장된 physical override 차이를 모니터 맵 안에서 바로 비교할 수 있게 한다.
- 새 모니터 연결/해제 시 “현재 저장값과 감지 결과 차이”를 배너나 카드로 알려준다.

완료 기준:
- 사용자가 stale inventory와 최신 inventory를 구분할 수 있다.
- 모니터 맵 편집 전에 현재 차이를 바로 이해할 수 있다.

### Track B. 노드 관리 심화와 재시작 플로우 (`Now -> Next`)

목표:
- GUI 노드 관리에서 아직 막아 둔 경로를 안전하게 확장한다.

핵심 방향:
- 이름 변경, 자기 자신 IP/포트 변경처럼 재시작이 필요한 작업은 별도 플로우로 분리한다.
- “즉시 반영 가능” 변경과 “저장 후 재시작 필요” 변경을 UI에서 명확히 나눈다.
- 삭제/추가/수정 이력과 영향 범위를 저장 직전에 더 분명히 확인시킨다.

완료 기준:
- 사용자가 어떤 변경이 즉시 반영되고 어떤 변경이 재시작이 필요한지 혼동하지 않는다.
- 위험한 자기 자신 변경이 안전한 흐름으로 가드된다.

### Track C. 설정 구조 운영성 강화 (`Next`)

목표:
- 분리된 설정 구조를 실제 운영에서 더 다루기 쉽게 만든다.

핵심 방향:
- 예제 설정도 `config.json / layout.json / monitor_overrides.json / monitor_inventory.json` 구조를 기준으로 정리한다.
- 처음 실행 시 필요한 파일이 없으면 기본 골격을 자동 생성하거나 더 분명한 안내를 준다.
- 설정 검사와 간단한 migration 보조 도구를 마련한다.

완료 기준:
- 새 사용자가 설정 파일 구조를 더 쉽게 이해한다.
- 예전 단일 config에서 새 구조로 넘어가는 경로가 더 친절해진다.

### Track D. 운영 UX 마감 (`Later`)

목표:
- 실사용 중 자잘한 마찰을 줄이고 관리 도구로서의 완성도를 높인다.

핵심 방향:
- 주요 작업의 undo/되돌리기 여지를 늘린다.
- 연결 상태, 감지 상태, 편집 잠금 상태를 더 일관된 배지 체계로 정리한다.
- 필요 시 tray / GUI / 로그 사이의 책임을 더 선명하게 다듬는다.

완료 기준:
- 자주 쓰는 운영 플로우가 더 짧고 덜 불안해진다.

## 5. 후순위 / 범위 밖 항목

- GUI 프레임워크 교체
- OS 디스플레이 설정을 앱이 직접 변경하는 기능
- 물리 배치 완전 자동 추론
- 인터넷 경유 노드 탐지와 계정 기반 동기화
- 대규모 테마 시스템

## 6. 작업 원칙

- 실제 장치 정보가 있으면 그것을 우선한다.
- UI 반응성은 과한 효과보다 즉시 상태 반영과 명확한 피드백으로 해결한다.
- 설정 분리는 파일 수 증가가 목적이 아니라 책임 경계 명확화가 목적이다.
- `ROADMAP.md`에는 완료 보고를 적지 않는다.
